## Approach: Skip Local Disk — Use Snowflake Native Connectors Directly

### Concept

Instead of the current flow:

```
MySQL → Local Disk (TSV/Parquet) → PUT to Stage → COPY INTO Snowflake
```

You eliminate local disk entirely:

```
MySQL → Python memory (batches) → Snowflake directly (via write_pandas / connector insert)
```

Or even better, for very large volumes:

```
MySQL → Snowflake via External Stage (S3/GCS/Azure) → COPY INTO
```

---

### Option A: `snowflake-connector-python` + `write_pandas()`

**How it works:**
- Read from MySQL in chunks using a server-side cursor (or connectorx with chunking)
- For each chunk, convert to a Pandas DataFrame
- Call `snowflake.connector.pandas_tools.write_pandas()` which internally does:
  - Compress the DataFrame to Parquet **in memory**
  - PUT it to a **temporary stage** (auto-created)
  - COPY INTO the target table
  - Clean up the temp stage

**Code pattern:**
```python
from snowflake.connector.pandas_tools import write_pandas

# Read MySQL in chunks (server-side cursor)
mysql_cursor = mysql_conn.cursor(dictionary=True)
mysql_cursor.execute("SELECT * FROM my_table WHERE updated_at > %s", (wm,))

while True:
    rows = mysql_cursor.fetchmany(500_000)
    if not rows:
        break
    df = pd.DataFrame(rows)
    write_pandas(sf_conn, df, table_name="MY_TABLE",
                 database="MY_DB", schema="RAW",
                 chunk_size=500_000, compression="snappy")
```

**Pros:**
- Zero local disk usage
- Memory bounded to one chunk at a time
- Built into the Snowflake connector (no external tooling)
- Handles Parquet creation + staging + COPY in one call

**Cons:**
- `write_pandas` does APPEND (INSERT) — no built-in MERGE. You'd still need a temp table + MERGE step for incremental.
- Slower than `mysqlsh` for multi-GB full loads (single-threaded compression)
- Requires `pandas` in memory (DataFrame overhead)

### Option B: `snowflake.connector` Bulk Insert via `executemany` / Binding

**How it works:**
- Use the connector's batch binding (`executemany` with `INSERT INTO ... VALUES (%s, ...)`)
- The connector auto-batches into internal stage files when row count is large

**Code pattern:**
```python
sf_cursor.executemany(
    "INSERT INTO MY_DB.RAW.MY_TABLE (COL1, COL2, ...) VALUES (%s, %s, ...)",
    list_of_tuples  # up to ~16K rows per batch
)
```

**Pros:**
- Simplest possible code
- No disk, no staging

**Cons:**
- Very slow for large volumes (row-by-row binding, even in batches)
- Not suitable for millions of rows
- No MERGE support

### Option C: Cloud Storage as External Stage (S3 / GCS / Azure Blob)

**How it works:**
- Extract from MySQL → write Parquet directly to cloud storage (using `pyarrow` + `s3fs` / `gcsfs` / `azure-storage-blob`)
- Snowflake reads from the external stage via `COPY INTO`
- No local disk involved — data goes straight to object storage

**Code pattern:**
```python
import s3fs
import pyarrow.parquet as pq

fs = s3fs.S3FileSystem()  # uses AWS credentials
arrow_table = cx.read_sql(mysql_uri, query, return_type="arrow")

# Write directly to S3 (no local file)
pq.write_table(arrow_table, "s3://my-bucket/staging/table/part_0001.parquet",
               filesystem=fs, compression="snappy")

# Then in Snowflake:
# COPY INTO DB.RAW.TABLE FROM @my_external_stage/ ...
```

**Pros:**
- Truly zero local disk
- Scales to TB+ (cloud storage is unlimited)
- Snowflake COPY FROM external stage is highly optimized (parallel micro-partitions)
- Can combine with chunked reads to also bound memory

**Cons:**
- Requires a cloud storage account (S3/GCS/Azure) + credentials
- Requires an external stage definition in Snowflake
- More infrastructure to manage (bucket policies, lifecycle rules)
- Adds cloud egress cost (MySQL → S3 → Snowflake)

### Option D: Snowpark `session.write_pandas()` / DataFrames

**How it works:**
- Use Snowpark Python to create a session
- Read MySQL chunks into Pandas DataFrames
- Use `session.write_pandas()` or `session.create_dataframe()` which internally stages data

**Code pattern:**
```python
from snowflake.snowpark import Session

session = Session.builder.configs(sf_config).create()
session.write_pandas(df, table_name="MY_TABLE",
                     database="MY_DB", schema="RAW",
                     auto_create_table=False, overwrite=False)
```

**Pros:**
- Same as Option A but with Snowpark's richer API
- Can chain transformations before writing
- Handles staging transparently

**Cons:**
- Same MERGE limitation (append only, need a temp + merge step)
- Snowpark session overhead

### Recommended Approach for Your Pipeline

**For incremental (MERGE) loads — Option A with a twist:**

```
MySQL (chunked read) → write_pandas() into TEMP table → MERGE INTO RAW
```

1. Read MySQL delta in chunks via server-side cursor (bounds memory)
2. `write_pandas()` each chunk into a transient/temp table (no local disk)
3. After all chunks are loaded into temp, run the same MERGE logic you already have

**For full loads — Option C (external stage) or Option A (write_pandas with TRUNCATE):**

- If you have cloud storage: write Parquet directly to S3/GCS → COPY INTO (fastest for large tables)
- If you don't: `TRUNCATE` target, then `write_pandas()` in chunks (slower but zero disk)

### What Changes in the Architecture

| Current | New (No Local Disk) |
|---------|-------------------|
| `mysqlsh` → local TSV → PUT → COPY | `write_pandas()` chunks → auto-stage → COPY (or external stage) |
| `connectorx` → local Parquet → PUT → COPY+MERGE | Server-side cursor → `write_pandas()` to temp → MERGE |
| Reconciler → local Parquet keys → PUT → COPY | `write_pandas()` key chunks to temp → anti-join |
| Needs GB of local disk | Needs only ~500MB–1GB RAM per worker |
| `mysqlsh` binary required | Pure Python (no external binary) |

---

### Trade-offs Summary

| Factor | Current (local disk) | write_pandas (no disk) | External stage (S3) |
|--------|---------------------|----------------------|-------------------|
| Disk requirement | High | None | None |
| Memory requirement | Low (stream to disk) | Medium (1 chunk in RAM) | Medium (1 chunk in RAM) |
| Speed (full, large) | Fast (mysqlsh parallel) | Slower (single-thread) | Fast (parallel COPY) |
| Speed (incremental) | Good | Similar | Similar |
| Infrastructure | mysqlsh binary + disk | Just Python | Cloud bucket + creds |
| Complexity | Medium | Low | Medium |
